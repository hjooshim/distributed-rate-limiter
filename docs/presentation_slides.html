Single Server — It Just Works
One server sees every request.
The counter is always accurate, so the limit is enforced correctly.
Multiple Servers — The Problem
Requests scatter across instances.
Each keeps its own counter — no server sees the full picture.
Distributed Rate Limiter — The Solution
A shared Redis store keeps the global counter.
Every instance checks the same state — limits are enforced consistently.

OOP · Encapsulation
Layer 1 — final + private fields
The fields are private and final — nobody can modify them after creation
public final class RateLimitDecision {
  private final boolean allowed;
  private final long retryAfterSeconds;
  ...
}
final class
Nobody can extend this — the result type is locked
private final fields
Set once in constructor — immutable after creation
OOP · Encapsulation
Layer 2 — private constructor
Cannot be instantiated via new directly — forces everyone through the factory
// The only way to build a RateLimitDecision
private RateLimitDecision(boolean allowed,
                        long retryAfterSeconds) {
  this.allowed = allowed;
  this.retryAfterSeconds = retryAfterSeconds;
}
private constructor
No one outside can call new RateLimitDecision() — guaranteed valid state
OOP · Encapsulation
Layer 3 — factory + validation
rejected() validates before creating — invalid states are never allowed into the system
// Reusable singleton — no allocation on happy path
private static final RateLimitDecision ALLOWED
  = new RateLimitDecision(true, 0);
public static RateLimitDecision allowed() {
  return ALLOWED; // single instance
}
public static RateLimitDecision rejected(long retryAfterSeconds) {
  if (retryAfterSeconds <= 0) throw new IllegalArgumentException(...); ← validates
  return new RateLimitDecision(false, retryAfterSeconds);
}
rejected() validates
Negative retry throws immediately — no invalid object can exist
allowed() = singleton
ALLOWED is reused — zero allocation on the happy path

Findings

Findings & Lessons Learned

🔗
Distributed Consistency
Maintaining accurate state across multiple nodes requires careful coordination — Redis Lua scripts ensure atomic read‑check‑update in a single round trip.
⚡
Race Condition Prevention
Atomic Redis operations eliminate race conditions that would otherwise allow limit breaches under concurrent traffic.
🛡
Safe Failure Behavior
When Redis is unreachable, the system must decide: fail open (allow) or fail closed (deny). This is a deliberate design choice.
🧩
Separation of Concerns Pays Off
Clean boundaries (AOP + Strategy Pattern) made the system easy to extend, test, and explain — adding a new algorithm means adding one class, no framework changes.
